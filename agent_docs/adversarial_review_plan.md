# Adversarial Review Plan — whatsapp-rust

> Generated by a pre-review understanding pass: 16 parallel subsystem mappers over the full
> ~226K-LOC workspace + a synthesis pass, cross-referenced against the captured WhatsApp-Web JS
> (`WASmaxIn*`: 1049 modules, `WASmaxOut*`: 436 — the wire-format ground truth).
>
> This document is the **organization** of the review: the classification groups, the ranked
> hotspots, and the 12 prioritized adversarial-review workflows. The workflows themselves are
> driven by the parameterized harness in `scripts/adversarial-review.workflow.js` (a copy is
> placed in the gitignored `.claude/workflows/adversarial-review.js` for name-based invocation).

## Executive summary

This is a mature, unusually well-annotated WhatsApp-Web reimplementation whose risk concentrates in three places: (1) the Signal crypto core and its caller-owned locking contract, (2) the durable-before-ack persistence invariant that spans the receive pipeline, connection lifecycle, and Signal store, and (3) app-state (syncd) LTHash accounting. The lowest layers (binary codec, Noise transport) are hardened but carry remote-DoS and trust-anchor hazards driven by untrusted server frames. The highest-severity issues are silent-data-loss and key/IV-reuse class: group/1:1 encryption does zero internal locking and relies entirely on callers holding the right lock, several persistence paths advance ratchet/version state before the durable row commits, and coordination locks live in capacity-bounded LRU caches whose eviction would mint duplicate locks. Compliance risk is real but tractable because captured-js is rich and already cross-referenced in comments; the sharp edges are phash byte-representation, KDF label/order exactness, and app-state anti-tampering ordering. Feature-gaps and overhead (unused failover paths, FIFO caches, sequential DB reads, stale MEX doc_ids) are genuine but separable into a lower-priority polish tier. VoIP is opt-in and isolated but ships a hand-ported codec on an unauthenticated recv path — a self-contained P0 fuzz target. Overall maturity is high; the danger is in a handful of hand-optimized shortcuts away from upstream (partial rollback, eviction-based key GC, non-transactional multi-write commits) where a single uncovered window is silent and unrecoverable.

## How this review is organized

**Priority scale**

| Priority | Meaning |
|---|---|
| **P0** | Correctness / security / data-loss / protocol-breaking — silent & unrecoverable class |
| **P1** | User-visible bugs, real perf/overhead wins, notable compliance gaps |
| **P2** | Polish, minor overhead, edge cases |
| **P3** | Nice-to-have |

**Classification groups**

| Group | Definition | Catches |
|---|---|---|
| **security** | Exploitable weaknesses reachable from an untrusted server/peer: DoS (crash/alloc/CPU), trust-anchor bypass, padding/nonce oracles, key-material disclosure, spoofed-protocol-message injection. | Unbounded decode recursion, cert-chain bypass, unauthenticated codec input, zero-key fallback, self-only gating bypass via DSM. |
| **concurrency** | Correctness under interleaving: lock scope covering the full read-modify-write, lock held across await/IO, cache-evictable locks, detached-task/generation fencing, semaphore/state-machine transitions. | Group vs DM session mutation collision, cache-backed lock eviction, drain→live commit races, dual teardown entry points, unlocked cache RMW. |
| **bug** | Logic/correctness defects independent of interleaving: off-by-one, error-branch matrices that neither ack nor retry, wrap-around math, silent None-vs-error swallowing, ordering assumptions. | Prekey watermark wrap, receive ack/retry matrix holes, poll tally ordering, offline-sync finisher never firing. |
| **compliance** | Byte/behavior deviation from WhatsApp-Web wire semantics, verified against captured-js ground truth and whatsmeow. | phash representation, KDF labels/order, MAC layout, token dictionary, anti-tampering ordering, ack-stanza shape, MEX doc_ids. |
| **feature-gap** | WA-Web capabilities parsed-but-unused or entirely absent that affect robustness or coverage. | Unused per-host failover, ignored usync backoff, undispatched notification types, missing newsletter admin ops, mediaretry stub. |
| **performance** | Latency/throughput on hot paths: sequential vs parallel IO, lock-held-across-network, per-message CPU amplification, redundant queries. | Sequential per-device DB reads, group_distribution_lock across network RTT, per-reaction metadata query, adversarial skip-ahead KDF cost. |
| **overhead** | Avoidable allocations/clones/redundant work with no correctness impact. | FIFO-vs-LRU eviction, per-command Device clone, unbounded lid_pn growth, StringHintCache linear scan, per-send Node clone. |

## Ranked hotspots (where the risk concentrates)

| # | Pri | Group | Subsystem — Area | Why |
|---|---|---|---|---|
| 1 | P0 | concurrency | crypto-signal — Caller-owned locking for group_encrypt/message_encrypt (zero internal locking; doc-only sender_key_lock contract) | A lock-scope bug lets two group_encrypts advance the same chain from the same iteration → identical CBC message key+IV → XOR of plaintexts. Catastrophic and invisible to round-trip tests. |
| 2 | P0 | concurrency | send-path — Cross-path Signal session mutation: SKDM fan-out ratchets per-device sessions under chain lock only; DM path guards same session under per-device lock | A device shared between an in-flight DM and a group SKDM target can be ratcheted by two different locks simultaneously → session desync → permanently undecryptable messages. |
| 3 | P0 | concurrency | receive-path — Offline-drain commit-batch drain→live transition + Signal-flush ordering (ReinsertGuard/epoch/generation state machine) | The invariant 'never persist a ratchet advance without a committed durable row' is protected by many interacting guards; one uncovered window turns a redelivery into an ackable duplicate = silent message loss. |
| 4 | P0 | concurrency | appstate — Per-patch persistence commits set_version BEFORE mutation-MAC store, no transaction (opposite of snapshot path) | A crash/error between version advance and MAC-store update permanently diverges the LTHash and wedges the collection into snapshot-MAC-mismatch retries. |
| 5 | P0 | compliance | store-persistence — Consumed-prekey crash-safe deletion atomicity (Present=delete/CheckedOut=defer/Absent=ask under one sessions-lock) | A wrong delete makes a redelivered pkmsg permanently undecryptable; a premature delete breaks forward secrecy via one-time-prekey double-use. |
| 6 | P0 | compliance | crypto-signal — KDF constants / X3DH agreement order / 8-byte MAC truncation / sender-key seed bytes vs captured-js | A single reordered DH or wrong HKDF label silently desyncs every session with real WA clients; captured-js ships the reference Signal library so this is directly diffable. |
| 7 | P0 | bug | lifecycle-connection — Offline-sync completion path: what actually calls complete_offline_sync (end-marker vs timeout) + semaphore widen 1→64 vs non-durable tail | If the normal end-marker path doesn't invoke the finisher, the client stays throttled at 1 permit forever; if the widen races the tail commit, messages are acked before Signal state is durable. |
| 8 | P0 | security | receive-path — Self-only protocol-message gating vs device-sent-message unwrap (has_invalid_dsm only warns) | A peer-spoofed DSM wrapper carrying key-share/history-sync/LID-mapping could bypass the is_from_me gate and poison app-state or the LID-PN cache. |
| 9 | P0 | security | noise-socket — Cert-chain verification bypass under cfg(test)/danger-skip feature; issuer_serial compared to hardcoded 0 with unwrap_or(0) | If the danger feature leaks into a real build via feature unification, a forged server static is silently accepted — full server impersonation. |
| 10 | P1 | concurrency | client-core-concurrency — Coordination locks (session_locks/chat_lanes/sender_key_locks) live in capacity-bounded LRU caches | Eviction of a live, referenced lock mints a second lock for the same key, silently breaking the per-sender/per-chat serialization the whole Signal layer depends on. |
| 11 | P0 | compliance | appstate — update_hash index-mode SET/REMOVE suppression interacting with process_patch in-patch overlay | The trickiest crypto-accounting logic; a mis-subtraction for SET+REMOVE-same-index orderings silently corrupts the LTHash and fails all future snapshot MACs. |
| 12 | P1 | security | binary-protocol — Unbounded decode recursion + speculative Vec::with_capacity from wire-controlled counts | A ~1MB nested-LIST frame drives ~340k recursion depth (stack overflow) and LIST_16/attribute pre-allocation up to 65535/32767 from untrusted bytes; reachable from the network if no upstream frame cap exists. |
| 13 | P1 | compliance | send-path — participant_list_hash (phash) byte-representation + failed-SKDM devices marked warm | phash drift causes redistribution churn or serves stale membership; marking encrypt-failed devices has_key can leave an alive device silently keyless if no retry receipt arrives. |
| 14 | P1 | security | crypto-signal — MessageKeyGenerator Serialized zero-key fallback ([0u8;32] with debug_assert only) | A fail-open KDF: if any path constructs Serialized bypassing from_pb length validation, a corrupt record yields an all-zero key in release. |
| 15 | P1 | security | voip — MlowDecoder::decode / on_rtp on the unauthenticated AES-CTR recv path | A hand-ported codec fed arbitrary relay bytes with no auth is the highest-severity remote-DoS surface; symmetric round-trip tests can't catch panics/over-alloc/infinite loops. |

## The 12 review workflows (prioritized & grouped)

### P0 tier

#### `review-signal-crypto-correctness` — Signal Crypto Core Correctness & KDF Compliance
- **group**: compliance  ·  **priority**: P0
- **scope**: crypto-signal: session_cipher.rs, ratchet.rs, ratchet/keys.rs, group_cipher.rs, sender_keys.rs, session.rs, protocol.rs, aes_gcm.rs, curve.rs, consts.rs
- **objective**: Confirmed byte-exact KDF/MAC/X3DH parity with captured-js and confirmed refutation-or-fix of: zero-key fallback reachability, skipped-key eviction vs replay boundary, deferred one-time-prekey double-use, partial-rollback correctness, self vs peer forward-jump misclassification, and hand-rolled GHASH correctness.
- **harness**: `compliance-diff(captured-js WA/Signal/Cipher.js,GroupCipher.js,LibraryConfig.js + whatsmeow + libsignal-upstream) -> KAT/NIST-vector cross-check for GCM -> find(lenses) -> dedup -> adversarial-verify by constructing the exact byte sequence / record that triggers each hypothesized fail-open -> synthesize`
- **captured-js refs**: GroupCipher.js, LibraryConfig.js + whatsmeow + libsignal-upstream
- **finder lenses**:
  - byte-level KDF label + DH-order + MAC-truncation diff vs captured-js
  - fail-open KDF paths (Serialized zero-key, all-zero accept)
  - skipped-key eviction re-derivation/replay after MAC-key cache drain
  - prekey lifecycle: consumed_prekey_id delete ordering vs session durability
  - MAC/signature-before-decrypt ordering on every branch (padding-oracle)
  - self-session classification unlocking the 25000-step ceiling
- **verify strategy**: Cross-check constants against three references (captured-js authoritative); write refuters that attempt to construct a Serialized variant bypassing from_pb, an evicted-then-replayed skipped key, and a peer classified as self; run GCM against NIST KAT beyond in-file tests; compile/test both simd and scalar builds.

#### `review-signal-session-concurrency` — Signal Session Locking Discipline & Lock-Cache Eviction
- **group**: concurrency  ·  **priority**: P0
- **scope**: send/group.rs, send/dm.rs, send/encrypt.rs, features/signal.rs, group_cipher.rs; client/adapters.rs, portable_cache.rs, handlers/message.rs; message_enqueue/session_locks
- **objective**: Prove (or produce a failing interleaving for) the two catastrophic guarantees: no two writers ever advance the same Signal/sender-key chain concurrently, and no live coordination lock can be evicted to mint a duplicate.
- **harness**: `find(lenses)->dedup->adversarial-verify by modeling concrete two-actor interleavings (group SKDM fan-out vs DM to shared device; concurrent group_encrypts; >cap distinct addresses forcing LRU eviction of a held lock)->synthesize`
- **finder lenses**:
  - cross-path session mutation: SKDM chain-lock vs DM per-device-lock on a shared device
  - group_encrypt load->advance->store atomicity under sender_key_lock scope
  - LRU eviction of a strongly-held Arc<Mutex> returning a fresh lock from get_with_by_ref
  - chat-lane worker lifecycle: evict-then-recreate spawning a second worker; enqueue into dead-worker channel
  - pkmsg_would_be_emitted take/restore session on retry paths without the main send lock
  - decrypt_group_message documented-unlocked vs concurrent group encrypt
- **verify strategy**: For each hypothesis construct the minimal concurrent scenario and show the shared-mutable-state access is unserialized; confirm whether the moka-style Cache can evict a key whose Arc is still strongly held and whether a subsequent get returns a different Arc (the core question); prefer a runnable stress repro over argument.

#### `review-durability-ack-ordering` — Durable-Before-Ack Invariant (Receive Commit Batch + Offline Sync + Prekey Store)
- **group**: concurrency  ·  **priority**: P0
- **scope**: message/commit_batch.rs, message/receive.rs (ack gating), message/durability.rs, client/sessions.rs, receipt.rs, store/signal_cache.rs (consumed-prekey), store/persistence_manager.rs (saver ordering)
- **objective**: Confirmed absence (or a concrete instance) of any path that acks/commits a stanza whose Signal/session/prekey state is only in cache, and of any drain→live transition that leaves the semaphore stuck or persists a rowless ratchet advance.
- **harness**: `trace-every-persistence-step -> crash/error-injection scenario construction (fail put_mutation_macs/store_session/flush at each step) -> adversarial-verify recoverability -> synthesize; compliance cross-ref MessageProcessorCache commit-batch ordering`
- **finder lenses**:
  - consumed-prekey durable/defer/ask decision + clear()-drops-buffer atomicity
  - commit_batch deferred-live-transition + ReinsertGuard epoch/generation gaps
  - offline-sync finisher: end-marker vs timeout path actually widening semaphore + flushing receipts only after tail durable
  - dual teardown entry points (cleanup_connection_state vs disconnect/reconnect) double-settle / lost clear
  - background saver dirty-flag vs snapshot-rebuild ordering on shutdown
- **verify strategy**: Inject failure at each individual backend write and check the collection/session/queue recovers vs permanently diverges; specifically resolve the open question of what calls complete_offline_sync on the normal path; treat any 'acked but cache-only' window as confirmed data-loss.

#### `review-appstate-sync-integrity` — App-State (syncd) LTHash, Anti-Tampering & MAC Compliance
- **group**: compliance  ·  **priority**: P0
- **scope**: appstate/processor.rs, hash.rs, lthash.rs, decode.rs, encode.rs, keys.rs, schemas.rs; appstate_sync.rs; client/app_state.rs; message/special.rs
- **objective**: Confirmed byte-exact MAC/key layout and anti-tampering accounting vs captured-js, plus resolution of the genesis-MAC-skip threat model and the fresh-pairing missing-key race.
- **harness**: `compliance-diff(WAWebSyncdAntiTampering, SyncdCollectionHandler, MutationKeyApi.Crypto, ValidateMutations, LtHash.js) -> find(lenses) -> adversarial-verify by exercising every SET/REMOVE ordering and a forged-aggregate-MAC genesis patch -> synthesize`
- **captured-js refs**: WAWebSyncdAntiTampering, SyncdCollectionHandler, MutationKeyApi.Crypto, ValidateMutations, LtHash.js
- **finder lenses**:
  - update_hash suppression vs process_patch in-patch overlay for all SET/REMOVE orderings
  - genesis had_no_prior_state aggregate-MAC skip abuse (server-forced empty reset + crafted baseline)
  - content/snapshot/patch MAC byte layout + AD-length packing vs captured
  - stale-snapshot / version-monotonicity gates vs 5-iter batched + 500-iter single loops
  - missing-key request/wait notifier race (false 'synced' cancelling watchdog)
  - SIMD vs scalar LTHash lane equivalence + pre-keyed HKDF-extract identity
- **verify strategy**: Table-drive every mutation ordering and diff net LTHash + persisted MAC store against WA-Web semantics; attempt a forged genesis patch under validate_macs=true; run KAT for MAC generators and SIMD/scalar parity across buffer sizes.

#### `review-receive-routing-recovery` — Receive Error-Recovery Matrix, Protocol-Message Gating & Inbound Routing
- **group**: bug  ·  **priority**: P0
- **scope**: message/receive.rs, retry.rs, dispatch.rs; message_processing.rs; protocol/retry.rs, nack.rs; handlers/router.rs, node_io.rs, notification/*, message.rs, ib.rs
- **objective**: Confirmed every session_outcome × group × bot combination either acks-or-retries exactly once, correct retry-reason/nack mapping vs captured-js, no self-only protocol message reachable from a spoofed DSM, and enumeration of undispatched/log-only inbound stanzas.
- **harness**: `exhaustive branch-matrix enumeration -> compliance-diff(DecryptionHandler, RetryRequest, NackFromStanza, HandleLoggedInStanza) -> adversarial-verify (spoofed-DSM injection attempt, unacked-and-unretried combo) -> synthesize`
- **captured-js refs**: DecryptionHandler, RetryRequest, NackFromStanza, HandleLoggedInStanza
- **finder lenses**:
  - ack/retry/nack matrix holes (stuck offline queue or double-ack)
  - spoofed device_sent_message smuggling key-share/history-sync/LID-mapping past is_from_me
  - retry-reason classification BadMac/InvalidPreKeyId/SessionNotFound vs SignalRetryable
  - chat-lane generation/enqueue-vs-worker-exit stranded message
  - notification type fall-through + mediaretry log-only stub + missing top-level <error> handler
  - detached-spawn cancellation inconsistency (prekey_low/identity-change/edge_routing unguarded) + non-serialized cache patches vs message worker
- **verify strategy**: Enumerate the outcome-flag cartesian product and trace each to a definite ack/retry terminal; pin nack codes against NackFromStanza.js; attempt to route a protocol message through a peer-authored DSM; grep captured-js HandleLoggedInStanza for the full top-level/notification case list and diff.

### P1 tier

- **captured-js refs**: WA/Wap/Dict.js, WAFrameSocket/ChatSocket/WANoise

## Recommended run order

1. **`review-signal-crypto-correctness`** — The crypto core is the foundation; KDF/MAC/rollback correctness must be pinned before reasoning about any locking or persistence built on top of it.
2. **`review-signal-session-concurrency`** — Depends on the crypto core's documented lock contract; the catastrophic key/IV-reuse and lock-eviction hazards gate the send and receive paths.
3. **`review-durability-ack-ordering`** — Consumes the session-locking conclusions; the durable-before-ack invariant is the top silent-data-loss surface and spans receive+lifecycle+store.
4. **`review-appstate-sync-integrity`** — Independent P0 crypto-accounting surface with its own non-transactional persistence risk; run in parallel with the durability track once crypto is pinned.
5. **`review-receive-routing-recovery`** — Builds on the durability and session-lock findings; the ack/retry matrix and spoofed-DSM gating are the inbound correctness+security keystone.
6. **`review-wire-transport-robustness`** — Lowest-layer hostile-frame robustness feeds every higher layer; can run in parallel early since it is largely self-contained (fuzz + captured-js diff).
7. **`review-send-path-compliance`** — After send-path concurrency is settled, verify its wire compliance (phash/namespace/warm-skip) that drives encryption targeting.
8. **`review-pairing-prekey-trust`** — Trust-root and prekey-pool correctness; self-contained and can run in parallel, but its prekey conclusions inform the crypto/store tracks.
9. **`review-connection-lifecycle-resilience`** — Depends on the durability track for the offline-sync completion conclusions; covers reconnect/keepalive/pull-loop resilience.
10. **`review-content-crypto-features`** — Higher-level content and media crypto; depends on the msg-secret and media-key conclusions but is lower blast-radius than the core.
11. **`review-voip`** — Opt-in and isolated; the P0 codec fuzz can run anytime in parallel, sequenced late because it affects only the voip feature build.
12. **`review-features-surface-overhead`** — Lowest-priority breadth/polish tier (feature-gaps, dead-code, overhead); run last once all correctness/security tracks are closed.

## How to run a workflow

The harness is parameterized; select a track by id. Portable form (tracked script):

```
Workflow({ scriptPath: 'scripts/adversarial-review.workflow.js', args: { id: 'review-signal-crypto-correctness' } })
Workflow({ scriptPath: 'scripts/adversarial-review.workflow.js', args: { ids: ['review-signal-crypto-correctness','review-appstate-sync-integrity'] } })
Workflow({ scriptPath: 'scripts/adversarial-review.workflow.js', args: { tier: 'P0' } })       // all P0 tracks
Workflow({ scriptPath: 'scripts/adversarial-review.workflow.js', args: { group: 'compliance' } })
Workflow({ scriptPath: 'scripts/adversarial-review.workflow.js', args: { all: true } })
```
When the copy is present in `.claude/workflows/`, the shorthand `Workflow({ name: 'adversarial-review', args: {...} })` also works.
Optional tuning args (all defaulted): `verifiers` (skeptics per finding, default 3), `maxFindingsPerTrack` (verify budget cap, default 20 — extra lower-priority candidates are logged, not silently dropped), `finderEffort`/`verifyEffort` (default `high`), `parallelTracks` (default false — tracks run sequentially since each already saturates the concurrency cap), `extraLenses` (append custom hunt angles).

Each run: fans out one finder agent per lens (with captured-js diffing where relevant) → dedups → adversarially verifies every finding with 3 independent skeptics (default-refute; a finding survives only on a majority "real" vote) → keeps only CONFIRMED findings → returns a structured report. Nothing is written to the repo by the review itself; the driver collects confirmed findings into a report file.

**Agent budget** ≈ `lenses + min(findings, maxFindingsPerTrack)×verifiers + 1` per track (roughly 30–70 agents). `all: true` stays under the 1000-agent-per-workflow backstop with the defaults, but is expensive as one invocation — prefer per-tier or per-track runs.
